# Milotic — Domain Glossary

## Actors

**Retail User** — non-technical retail investor or trader. Primary audience for v1. Interacts exclusively via Claude Desktop / claude.ai. Never writes code or manages CLI commands.

**Mobile App** — the broker's existing authenticated mobile trading app. Trusted device for QR-based session linking.

**MCP Server** (Milotic) — the FastMCP server that wraps the Go backend. Exposes trading operations as tools. Advisory layer only — never authoritative for safety or compliance.

**Go Backend** — the authoritative trading API. Owns all business logic, execution, limits, audit logging, and order status. Treats every call from the MCP layer as untrusted input.

---

## Authentication

**SSO** — Single Sign-On flow via the broker's login microsite. Produces a `subAccountId` and `auth_token` used for all subsequent API calls.

**QR Session** — a short-lived pairing token generated by the backend (`POST /auth/qr-session`). Displayed as a QR image by `system_connect_start`. Confirmed when the Retail User scans it with the Mobile App (`POST /auth/qr-confirm` on mobile). Expires after ~120 seconds.

**auth_token** — session credential obtained after QR confirmation. Sent as a header on every API call. Lifecycle: per-day (expires at market close). Stored in FastMCP session state via `ctx.set_state("auth_token", ...)`.

**subAccountId** — identifies the specific sub-account to trade under. Sent as a header on every API call. Resolved from the QR auth flow and stored in `ctx.set_state("sub_account_id", ...)`.

**Session State** — per-session key-value store managed by FastMCP's `ctx.set_state()` / `ctx.get_state()`. Keyed by session ID (UUID for stdio). TTL: 86400 seconds (24h), matching the per-day auth_token lifecycle.

---

## Order Lifecycle

**Order Intent** — the Retail User's expressed desire to buy or sell, as received by the MCP layer. Treated as untrusted input by the backend.

**Order Preview** — summary of an Order Intent returned when `confirmed=False`. No order is placed. Contains symbol, side, quantity, order_type, price, and a prompt for confirmation.

**Order Placement** — actual submission to the backend, triggered when `trading_place_order` is called with `confirmed=True`. Generates a Reference ID.

**Reference ID** — a UUID generated by the MCP layer and sent with every `trading_place_order` call. Stored by the backend. Used to look up order status after a timeout. Not an idempotency key — the backend treats every call as a unique order.

**Unknown Outcome** — the state after a network timeout following order submission. True execution status is unknown. MCP informs the Retail User and surfaces the Reference ID for manual reconciliation via `trading_get_order`. Never retried automatically.

**Partial Fill** — a legitimate order state where only some quantity was executed. Not an error. Reported as-is to the Retail User.

---

## Consent

**Per-Trade Consent** — default consent model for v1. Retail User must explicitly confirm each order by calling `trading_place_order` a second time with `confirmed=True`. The first call always returns an Order Preview.

**Day-Pass Consent** — deferred to v2. An optional bounded consent envelope allowing execution without per-trade approval, capped by order count + total notional, expiring at market close.

---

## Checks

**Advisory Check** — a check at the MCP layer for fast UX feedback. Fail-open: if uncertain, passes through to the backend. Never a security or compliance boundary. Examples: symbol normalisation, side/order_type validation, positive quantity check.

**Authoritative Check** — a check at the backend layer that is always enforced. Fail-closed. Examples: position limits, balance checks, rate limits, kill switch, audit log.

---

## Tools

**system_connect_start** — initiates QR authentication. Returns a QR image (base64) and a session_id. No state is set until `system_connect_verify` succeeds.

**system_connect_verify** — polls the backend until the QR session is confirmed by the Mobile App (up to 120s timeout). Reads `qr_session_id` from session state (set by `system_connect_start`); the client does not pass it as a parameter. On success, sets `auth_token` and `sub_account_id` in session state.

---

## Deferred (v2)

- **Day-Pass Consent** — optional bounded daily consent envelope
- **Basket Order** — atomic multi-leg order placement (`trading_place_basket`)
- **Advisory Rate Throttle** — MCP-layer early rejection before the 10 orders/sec compliance cap
